# v0.5.2 Patch Release Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix the `__version__` drift surfaced by the v0.5.1 pre-prod review (L1) — `eap_core.__version__` and `eap_cli.__version__` return `"0.2.0"` while `pyproject.toml` says `"0.5.1"`. The drift has accumulated silently across every release since v0.2.0 because the `_version.py` files were never bumped alongside `pyproject.toml`. Ship as v0.5.2.

**Architecture:** Replace hardcoded `__version__ = "X.Y.Z"` with `importlib.metadata.version(<package>)` so there's exactly one source of truth (the install metadata, derived from `pyproject.toml`). Future version bumps to `pyproject.toml` automatically propagate to `__version__` without anyone having to remember a second file. Plus a regression test that pins `__version__ == pyproject.toml::project.version` so this drift class cannot recur silently.

**Tech Stack:** Python 3.11+ (`importlib.metadata` is stdlib). No new dependencies. No public API change (`__version__` was already a public attribute; the value it returns changes from stale to live).

**Scope deferred to v0.6.0:** L2, L3, N1, N2, N3 from the v0.5.1 review (all minor housekeeping) plus the existing v0.5.0 carryovers (M-N1 through M-N6, L-N1 through L-N6, N-N1 through N-N3) plus the original v0.4.0 deferred Highs.

## Task 1: Fix `__version__` drift permanently

**Files:**
- Modify: `packages/eap-core/src/eap_core/_version.py`
- Modify: `packages/eap-cli/src/eap_cli/_version.py`
- Create: `packages/eap-core/tests/test_version.py`

The fix is to derive `__version__` at import time from the installed package metadata rather than hardcoding it. This makes `pyproject.toml` the single source of truth and is the idiomatic Python pattern (see [PEP 396](https://peps.python.org/pep-0396/) successor practice and the `importlib.metadata` stdlib docs).

- [ ] **Step 1.1: Failing test — `__version__` matches `pyproject.toml`**

Create `packages/eap-core/tests/test_version.py`:

```python
"""Pin __version__ to the pyproject.toml::project.version.

Without this test, the two source-of-truth values can drift silently
across releases (as they did between v0.2.0 and v0.5.1 — see L1 of the
v0.5.1 pre-prod review).
"""

from __future__ import annotations

import tomllib
from pathlib import Path

import pytest

_REPO_ROOT = Path(__file__).resolve().parents[3]


def _read_pyproject_version(package: str) -> str:
    pyproject = _REPO_ROOT / "packages" / package / "pyproject.toml"
    if not pyproject.is_file():
        pytest.skip(f"pyproject.toml for {package} not present at {pyproject}")
    data = tomllib.loads(pyproject.read_text(encoding="utf-8"))
    return data["project"]["version"]


def test_eap_core_version_matches_pyproject():
    import eap_core
    expected = _read_pyproject_version("eap-core")
    assert eap_core.__version__ == expected, (
        f"eap_core.__version__ ({eap_core.__version__!r}) drifted from "
        f"pyproject.toml::project.version ({expected!r}). "
        f"Either bump _version.py or migrate to importlib.metadata."
    )


def test_eap_cli_version_matches_pyproject():
    import eap_cli
    expected = _read_pyproject_version("eap-cli")
    assert eap_cli.__version__ == expected, (
        f"eap_cli.__version__ ({eap_cli.__version__!r}) drifted from "
        f"pyproject.toml::project.version ({expected!r}). "
        f"Either bump _version.py or migrate to importlib.metadata."
    )
```

Note: `tomllib` is stdlib in Python 3.11+. The workspace already pins `requires-python = ">=3.11"`.

Note: the test asserts `eap_cli.__version__` exists. Verify that `packages/eap-cli/src/eap_cli/__init__.py` re-exports `__version__` from `_version.py`. If it doesn't, this test will fail with `AttributeError` first; the fix in step 1.4 will need to add the re-export.

- [ ] **Step 1.3: Switch `eap_core/_version.py` to `importlib.metadata`**

Replace the entire contents of `packages/eap-core/src/eap_core/_version.py`:

```python
"""Single source of truth for eap-core's version.

Reads from the installed package metadata so `pyproject.toml::project.version`
is the canonical value. Without this indirection, _version.py would have to
be bumped alongside every pyproject.toml change — and historically it wasn't
(see v0.5.1 pre-prod review finding L1).

If the package isn't installed (e.g. raw source clone without `pip install -e`
or `uv sync`), `__version__` falls back to ``"unknown"``. The fallback is
distinguishable from any real version string so audit/observability consumers
can detect the dev-tree case.
"""

from __future__ import annotations

from importlib.metadata import PackageNotFoundError, version

try:
    __version__: str = version("eap-core")
except PackageNotFoundError:
    __version__ = "unknown"
```
