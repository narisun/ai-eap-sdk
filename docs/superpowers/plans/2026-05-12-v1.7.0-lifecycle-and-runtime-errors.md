## Task 1: `on_call_end` lifecycle hook for unary path

**Goal:** Add a `finally`-block hook to `MiddlewarePipeline.run` mirroring v1.6.2's `on_stream_end` on `run_stream`. Strict-additive: `PassthroughMiddleware` provides a no-op default.

**Files:**
- Modify: `packages/eap-core/src/eap_core/middleware/base.py` — add `on_call_end` to `Middleware` Protocol + `PassthroughMiddleware`
- Modify: `packages/eap-core/src/eap_core/middleware/pipeline.py` — wire `on_call_end` into `run()`
- Modify: `packages/eap-core/tests/test_pipeline.py` — add no-op `on_call_end` to existing direct-Protocol mocks (`RecordingMiddleware`, `Tagger`, `BoomStream`)
- Test: `packages/eap-core/tests/test_middleware_call_lifecycle.py` (new)

**Design:**

Today `MiddlewarePipeline.run()`:
- `on_request` (L→R)
- `terminal(req, ctx)`
- `on_response` (R→L) **only on success**
- `on_error` (R→L) **only on failure**

After T1:
- `on_request` (L→R) (unchanged)
- `terminal(req, ctx)` (unchanged)
- `on_response` (R→L) **only on success** (unchanged)
- `on_call_end` (R→L) **always**, in `finally` block
- `on_error` (R→L) **only on failure** (unchanged)

Order on exception: `on_call_end` fires BEFORE `on_error` (inner `finally` runs before outer `except`). Same semantics as `on_stream_end` on the streaming path.

Errors in `on_call_end` are best-effort — logged via `_LOG.warning(..., exc_info=secondary)`, do not mask primary terminal exceptions.

- [ ] **Step 4: Wire `on_call_end` into `run()`**

Edit `MiddlewarePipeline.run()`:

```python
async def run(self, req: Request, ctx: Context, terminal: Terminal) -> Response:
    ran: list[Middleware] = []
    try:
        try:
            for mw in self._mws:
                ran.append(mw)
                req = await mw.on_request(req, ctx)
            resp = await terminal(req, ctx)
            for mw in reversed(ran):
                resp = await mw.on_response(resp, ctx)
            return resp
        finally:
            # on_call_end runs right-to-left, best-effort, in a finally
            # block so it fires on BOTH normal completion AND failure —
            # mirrors on_stream_end semantics from run_stream().
            for mw in reversed(ran):
                try:
                    await mw.on_call_end(ctx)
                except Exception as secondary:
                    mw_name = getattr(mw, "name", type(mw).__name__)
                    _LOG.warning(
                        "secondary failure in %s.on_call_end during call finalization",
                        mw_name,
                        exc_info=secondary,
                    )
    except Exception as exc:
        await self._on_error(ran, exc, ctx)
        raise
```

Note the **nested try/try/finally/except** structure: the inner `try` covers the request→terminal→response chain plus the cleanup `finally`; the outer `try/except` catches anything raised from there (including from `on_call_end`'s primary execution, which we DON'T want — but `on_call_end`'s secondary errors are already swallowed by their own inner try/except). The terminal exception propagates from `await terminal(...)` → through the inner `finally` (where `on_call_end` runs and logs any secondaries) → out to the outer `except`, where `_on_error` runs and the exception re-raises.

## Task 2: PII finalization — buffer flush + vault cleanup

**Goal:** Wire PII's existing per-context state through the new lifecycle hooks.

**Files:**
- Modify: `packages/eap-core/src/eap_core/middleware/pii.py`
- Test: `packages/eap-core/tests/test_pii_finalization.py` (new)

**Design:**

Two real bugs land here. Both flagged in v1.6.2 CHANGELOG as `Follow-up (v1.7 backlog)`:

1. **PII stream buffer leak on mid-stream exception.** `pii.py` keeps `ctx.metadata["pii._stream_buffer"]` to handle partial vault tokens that straddle chunk boundaries. Today the buffer flushes only when a chunk arrives with `finish_reason` set. If upstream raises mid-stream, the buffer leaks held text — operator never sees the trailing fragment, and the next reader of `ctx.metadata` (e.g., an audit middleware) sees stale buffer contents.

2. **PII vault cleanup on call end.** The vault (`ctx.metadata["pii._vault"]` or wherever it lives — verify) holds the mapping from `<LABEL_xxxx>` tokens to original PII strings. Today nothing clears it; ctx GC handles cleanup in practice, but middlewares that retain ctx (e.g., for async background work, or via `_owned_components` patterns) leak the vault. Symmetric with T1's "always-fire on_call_end" gives the right hook.

After T2:
- PII overrides `on_stream_end` to flush the per-context buffer to a final emission (or discard if mid-exception) AND clear `ctx.metadata["pii._stream_buffer"]`.
- PII overrides `on_call_end` to clear `ctx.metadata["pii._vault"]` (or whichever key actually stores the vault) AND `ctx.metadata["pii._stream_buffer"]` (defense-in-depth).

```python
async def on_stream_end(self, ctx: Context) -> None:
    # Flush any held partial chunk; clear the buffer either way.
    buffer = ctx.metadata.pop(_STREAM_BUFFER_KEY, None)
    if buffer:
        # Behavior decision: re-emit on normal completion? Or just clear?
        # Re-emitting would require yielding from on_stream_end, which the
        # Protocol doesn't allow. So clear-only: a buffer that survives the
        # final chunk indicates the upstream stopped abruptly with a
        # partial vault token — emit a WARNING log so operators notice.
        _LOG.warning(

            "upstream finished without a finish_reason chunk. Held text "
            "is being dropped to prevent state leak across requests.",
            len(buffer),

async def on_call_end(self, ctx: Context) -> None:

    ctx.metadata.pop(_VAULT_KEY, None)
    ctx.metadata.pop(_STREAM_BUFFER_KEY, None)
```

New module `runtimes/errors.py`:

```python
"""Canonical runtime exception hierarchy.

Adapters MUST translate vendor exceptions into one of these types so
middleware ``on_error`` handlers can react uniformly regardless of
provider (Bedrock, Vertex, OpenAI, Anthropic, local).
"""

from __future__ import annotations


class RuntimeAdapterError(Exception):
    """Base exception for all runtime adapter failures.

    All adapter-raised exceptions inherit from this. Mid-pipeline
    middleware can `except RuntimeAdapterError:` to catch any
    provider-level failure without coupling to a specific vendor SDK.
    """


class RuntimeAuthError(RuntimeAdapterError):
    """Authentication or authorization failure with the provider.

    Vendor mappings: botocore.exceptions.ClientError with
    code in {AccessDeniedException, UnauthorizedException};
    google.api_core.exceptions.PermissionDenied; HTTP 401/403.
    """


class RuntimeRateLimitError(RuntimeAdapterError):
    """Provider returned a rate-limit or quota error.

    Caller may retry with backoff. Vendor mappings: ThrottlingException,
    google.api_core.exceptions.ResourceExhausted, HTTP 429.
    """


class RuntimeTimeoutError(RuntimeAdapterError):
    """Request exceeded the provider's timeout window."""


class RuntimeServerError(RuntimeAdapterError):
    """Provider returned a 5xx / unavailable / transient server error.

    Caller may retry with backoff.
    """


class RuntimeContextLengthError(RuntimeAdapterError):
    """Request exceeded the model's context window.

    Caller should reduce prompt size; retry will not help.
    """
```

`BaseRuntimeAdapter` docstring updates to document the contract:

```python
class BaseRuntimeAdapter(ABC):
    """Vendor-neutral runtime adapter ABC.

    Exception contract: implementations MUST translate vendor-specific
    exceptions into the canonical types in :mod:`eap_core.runtimes.errors`.

    subclass — keep the original exception accessible via ``__cause__``
    so audit logs can inspect the vendor payload.
    """
```

```python
def _map_botocore_error(exc: Exception) -> RuntimeAdapterError:
    """Map botocore.exceptions to canonical EAP-Core runtime errors."""
    try:
        from botocore.exceptions import ClientError
    except ImportError:
        return RuntimeAdapterError(str(exc)) from exc  # type: ignore[misc]

    if isinstance(exc, ClientError):
        code = exc.response.get("Error", {}).get("Code", "")
        if code in {"AccessDeniedException", "UnauthorizedException"}:
            return RuntimeAuthError(str(exc))
        if code in {"ThrottlingException", "TooManyRequestsException"}:
            return RuntimeRateLimitError(str(exc))
        if code in {"ServiceUnavailableException", "InternalServerError"}:
            return RuntimeServerError(str(exc))
        if code == "ValidationException" and "context" in str(exc).lower():
            return RuntimeContextLengthError(str(exc))
    return RuntimeAdapterError(str(exc))
```

Vertex adapter mapping similar, using `google.api_core.exceptions`.

Each real-runtime call site wraps with `try / except / raise _map_*_error(exc) from exc`. The `from exc` preserves `__cause__` so audit can see the underlying vendor payload.

```python
class ThreatDetectionMiddleware(PassthroughMiddleware):

    service). Default detector is :class:`RegexThreatDetector` for

        min_confidence: float = 0.7,
        block: bool = True,
```

### 6a — Luhn validation for credit-card PII (P1-6)

Today PII's CREDIT_CARD regex matches any 16-digit number with spaces/dashes — false positives are real (`1234 5678 9012 3456` is not a valid card). Add a Luhn modulus-10 check that filters non-card matches before masking:

```python
def _passes_luhn(card_number: str) -> bool:
    digits = [int(c) for c in card_number if c.isdigit()]
    if len(digits) not in (13, 14, 15, 16, 19):
        return False
    checksum = 0
    for i, d in enumerate(reversed(digits)):
        if i % 2 == 1:
            d *= 2
            if d > 9:
                d -= 9
        checksum += d
    return checksum % 10 == 0
```

Wire it into the credit-card regex match path. Tests with known-valid and known-invalid 16-digit sequences.

```python
class LocalRuntimeAdapter(BaseRuntimeAdapter):
    def __init__(self, config: RuntimeConfig) -> None:
```

### 6e — Buffering contract documentation (Finding 6)

Add a section to `docs/developer-guide.md`:

```markdown
### Buffering middleware contract (streaming path)

Some middlewares need to buffer stream chunks before emission — PII
vault detokenization across token boundaries is the canonical example.
The contract:

1. A buffering middleware may consume `Chunk` objects in `on_stream_chunk`
   without emitting an equivalent `Chunk`. Return a `Chunk(text="")` with
   the same `index` so downstream middlewares see a clean pass-through.
2. The buffering middleware MUST emit any held buffer by `on_stream_end`
   at the latest. The pipeline does NOT scan chunk-emit events; it relies
   on the middleware to drain.
3. On exception (terminal raises mid-stream), the buffering middleware's
   `on_stream_end` MUST clear its buffer. Re-emission of partial content
   is a security/correctness liability and must be opt-in per middleware.
```
